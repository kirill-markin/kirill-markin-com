# Codex के नियम: वैश्विक निर्देश, AGENTS.md और Mac ऐप

मैंने Codex का गंभीर इस्तेमाल अभी करीब एक महीने पहले ही शुरू किया है, खासकर तब से जब GPT-5.4 ने इसे मेरे लिए सच में उपयोगी बना दिया। इसलिए यह कोई "पांच साल की कसौटी पर परखी हुई समझ" वाला लेख नहीं है। मेरी बात इससे कहीं सरल है: Codex मेरे लिए तभी बेहतर हुआ, जब मैंने उसे एक और चमकीली चैट-विंडो की तरह लेना बंद किया और शुरुआत से ही उसे स्थिर निर्देश देने शुरू किए।

OpenAI के पास Codex CLI भी है और Mac ऐप भी। मेरे लिए दोनों की बुनियाद एक ही है: स्थायी निर्देश, `AGENTS.md`, रिपॉज़िटरी के नियम, और ऐसा एजेंट जो मेरी काम करने की शैली पहले से लेकर चले। मैं Mac ऐप को इसलिए प्राथमिकता देता हूं, क्योंकि उसमें काम करना एक और टर्मिनल विंडो में अटके रहने की तुलना में कहीं अधिक सहज लगता है।

Codex CLI यह काम पहले से अच्छी तरह करता है। Mac ऐप उसी कामकाज को बस अधिक साफ और सुखद रूप देता है। हर नए काम में मैं बार-बार वही बातें नहीं सिखाना चाहता: सख्त typing, न्यूनतम बदलाव, स्पष्ट त्रुटियां, बेवजह के fallback नहीं, और बिखरी हुई व्याख्याओं की जगह code में docstrings। मैं चाहता हूं कि यह आधार शुरुआत से मौजूद रहे।

व्यवहार में यह सब `Settings -> Personalization -> Custom instructions` में रहता है।

भीतर की व्यवस्था यह है कि ऐप के ये निर्देश व्यक्तिगत `AGENTS.md` में जाकर दर्ज होते हैं। यह मुझे ठीक लगता है। इससे एक ओर बेहतर ऐप अनुभव मिलता है, और दूसरी ओर CLI जैसी फ़ाइल-आधारित स्पष्टता भी बनी रहती है।

## Codex Custom Instructions वास्तव में कहाँ रहते हैं

अगर इस लेख से आपको सिर्फ एक स्क्रीन याद रखनी हो, तो यही होनी चाहिए।

ऐप में वैश्विक निर्देश `Settings -> Personalization -> Custom instructions` के भीतर रहते हैं। मैं सबसे पहले यही स्क्रीन दिखाऊंगा।

OpenAI के [Codex दस्तावेज़](https://developers.openai.com/codex/) बताते हैं कि Codex आपके Codex home directory में रखी वैश्विक निर्देश-फ़ाइल पढ़ सकता है, जो सामान्यतः `~/.codex/AGENTS.md` होती है। [Codex settings दस्तावेज़](https://developers.openai.com/codex/config/) यह भी कहते हैं कि custom instructions में किया गया बदलाव व्यक्तिगत `AGENTS.md` को अपडेट करता है।

मुझे यही ढांचा चाहिए। मैं ऐप को अपना मुख्य इंटरफ़ेस बना सकता हूं, और फिर भी उसके पीछे मौजूद असली फ़ाइल की स्पष्टता नहीं खोता।

मेरे मन में इसकी परतें कुछ इस तरह बैठती हैं:

1. अलग-अलग परियोजनाओं में लागू मेरी डिफ़ॉल्ट प्राथमिकताओं के लिए व्यक्तिगत `~/.codex/AGENTS.md`
2. टीम और रिपॉज़िटरी से जुड़े निर्देशों के लिए रिपॉज़िटरी `AGENTS.md`
3. इन नियमों के ऊपर बैठी Codex ऐप की सेटिंग्स और रिपॉज़िटरी मार्गदर्शन

## वे नियम जिन्हें मैं चाहता हूं कि Codex हर जगह पहले से लेकर चले

यह वह आधार है जिसे मैं चाहता हूं कि Codex किसी भी रिपॉज़िटरी में परियोजना-विशेष निर्देश देखने से पहले साथ लेकर आए।

```markdown
# व्यक्तिगत AGENTS.md

## कोड शैली

- टिप्पणियां केवल English में हों
- OOP की तुलना में functional programming को प्राथमिकता दें
- OOP classes का उपयोग केवल connectors और external systems के interfaces के लिए करें
- pure functions लिखें - केवल return values बदलें, input parameters या global state नहीं
- DRY, KISS और YAGNI सिद्धांतों का पालन करें
- हर जगह strict typing रखें - function returns, variables, collections
- नया code लिखने से पहले जांच लें कि वही logic पहले से मौजूद तो नहीं है
- untyped variables और generic types से बचें
- किसी parameter का default value न दें - हर parameter को स्पष्ट रखें
- जटिल data structures के लिए सही type definitions बनाएं
- सभी imports फ़ाइल की शुरुआत में हों
- सरल, एक ही काम करने वाले functions लिखें - multi-mode behavior नहीं, और ऐसे flag parameters नहीं जो logic बदल दें

## त्रुटि-प्रबंधन

- errors को हमेशा स्पष्ट रूप से raise करें, उन्हें चुपचाप अनदेखा न करें
- ऐसे specific error types इस्तेमाल करें जो साफ बताएं कि क्या गलत हुआ
- ऐसे catch-all exception handlers से बचें जो मूल कारण छिपा दें
- error messages साफ और काम के होने चाहिए
- fallback तभी रखें जब मैं उसे स्पष्ट रूप से मांगूं
- लक्षण नहीं, मूल कारण ठीक करें
- external API या service calls में retries और warnings का उपयोग करें, फिर आखिरी error raise करें
- error messages में debugging के लिए पर्याप्त संदर्भ हो: request params, response body, status codes
- logging में dynamic values को message strings में पिरोने के बजाय structured fields का उपयोग करें

## टूलिंग और निर्भरताएं

- `pyproject.toml` और `package.json` जैसी आधुनिक package management files को प्राथमिकता दें
- dependencies को global रूप से नहीं, project environments में install करें
- dependencies को project config files में जोड़ें, अलग से एक-बार वाले installs के रूप में नहीं
- जरूरत पड़ने पर installed dependencies का source code पढ़ें, behavior का अनुमान न लगाएं

## परीक्षण

- रिपॉज़िटरी की मौजूदा testing strategy और existing test suite का सम्मान करें
- default रूप से नए unit tests न जोड़ें
- जब tests की जरूरत हो, तो integration, end-to-end या smoke tests को प्राथमिकता दें जो वास्तविक behavior को परखें
- unit tests का उपयोग कम ही करें, मुख्यतः stable datasets या pure data transformations के लिए
- केवल coverage numbers बढ़ाने के लिए unit tests कभी न जोड़ें
- जहां real calls व्यावहारिक हों, वहां mocks से बचें
- अक्सर mock-based coverage बनाए रखने की तुलना में real API या service calls पर थोड़ा खर्च करना बेहतर होता है
- मांगे गए बदलाव के लिए जितनी जरूरत हो, उतना ही न्यूनतम test coverage जोड़ें

## Codex कार्यप्रवाह

- edit करने से पहले रिपॉज़िटरी को समझें
- अनुमान लगाने से पहले सक्रिय `AGENTS.md` files पढ़ें
- changes को न्यूनतम और मौजूदा request से सीधे जुड़ा हुआ रखें
- रिपॉज़िटरी की मौजूदा शैली से मेल रखें, भले ही वह मेरी व्यक्तिगत पसंद से अलग हो
- unrelated changes को revert न करें
- code search के लिए `rg` को प्राथमिकता दें
- flags वाले non-interactive commands का उपयोग करें
- हमेशा non-interactive git diff का उपयोग करें: `git --no-pager diff` या `git diff | cat`
- code changes के बाद वही relevant tests या validation commands चलाएं जिन्हें project पहले से परिभाषित करता है

## दस्तावेज़ीकरण

- code ही मुख्य documentation है - साफ naming, types, और docstrings का उपयोग करें
- documentation को उन्हीं functions या classes की docstrings में रखें जिनका वह वर्णन करती है, अलग files में नहीं
- अलग docs files तभी बनाएं जब कोई concept code में साफ व्यक्त न हो सकता हो
- documentation को अलग-अलग files में दोहराएं नहीं
- जानकारी को वर्तमान स्थिति के रूप में रखें, बदलावों की changelog की तरह नहीं
```

यह फ़ाइल अपने सबसे अच्छे अर्थ में नीरस है।

यह बार-बार होने वाली झुंझलाहटें हटाने के लिए है:

- जब मुझे छोटा सा patch चाहिए हो, तब Codex का बेवजह बड़ा refactor गढ़ देना
- Codex का अपनी अनिश्चितता को मुलायम भाषा के पीछे छिपा देना
- "सुरक्षा" के नाम पर fallback behavior जोड़ देना
- prompt बहुत संकरा होने पर रिपॉज़िटरी की परंपराओं को अनदेखा कर देना

जब ये नियम पहले से लोड होते हैं, तो पूरा सत्र कहीं अधिक शांत और भरोसेमंद हो जाता है।

## मैं वैश्विक नियमों को रिपॉज़िटरी नियमों से ऊपर क्यों रखता हूं

Codex में `AGENTS.md` की परतदार व्यवस्था का अच्छा समर्थन है। OpenAI global file `~/.codex/AGENTS.md` का ज़िक्र करता है, और फिर working directory जितनी अधिक विशिष्ट होती जाती है, उतनी ही रिपॉज़िटरी और nested directory files लागू होती जाती हैं।

यह परतदार व्यवस्था उपयोगी है, लेकिन पहली परत फिर भी मेरी होनी चाहिए।

मेरी व्यक्तिगत फ़ाइल को इन सवालों का जवाब देना चाहिए:

- मैं कोडिंग शैली को कितना सख्त रखना चाहता हूं
- मेरे लिए स्वीकार्य त्रुटि-प्रबंधन क्या है
- बदलाव करते समय एजेंट कितना आक्रामक हो
- सामान्य कोडिंग काम में "done" का मतलब क्या है

रिपॉज़िटरी फ़ाइल को इन सवालों का जवाब देना चाहिए:

- यह codebase किस तरह व्यवस्थित है
- कौन से commands चलाने हैं
- कौन से हिस्से नाजुक हैं
- टीम PRs, commits या दस्तावेज़ों को कैसे संभालना चाहती है

संक्षेप में:

- व्यक्तिगत `AGENTS.md` बताता है कि मैं कैसे काम करता हूं
- रिपॉज़िटरी `AGENTS.md` बताता है कि यह codebase कैसे चलती है

अगर मैं दोनों को मिला दूं, तो दोहराव होगा, धीरे-धीरे अंतर पैदा होगा, और आखिर में ऐसी फ़ाइल बनेगी जिसे कोई संभालना नहीं चाहेगा।

यही वजहों में से एक है कि Codex मेरे लिए उम्मीद से बेहतर निकला। इसकी निर्देश-श्रृंखला साफ दिखाई देती है। यह किसी छिपे हुए prompt-जुगाड़ जैसा कम और एक वास्तविक तंत्र जैसा ज़्यादा लगता है।

## Mac ऐप ही मेरा मुख्य इंटरफ़ेस है, और यह मायने रखता है

इस समय Codex Mac ऐप का हिस्सा मुझे सबसे ज़्यादा पसंद आ रहा है।

यह इसलिए नहीं कि CLI कमजोर है। CLI पहले से बहुत अच्छा है। ऐप बस रोज़मर्रा के इस्तेमाल के लिए अधिक सुखद है। नीचे वही Codex है, ऊपर कहीं बेहतर अनुभव।

इसीलिए मैं इस लेख का केंद्र CLI को नहीं बना रहा, भले ही CLI महत्वपूर्ण हो। उसी तंत्र को इस्तेमाल करने का मेरे लिए बेहतर तरीका ऐप है।

ऐप को केवल सजावटी चीज़ की जगह एक ठोस साधन बनाने वाली बात यह है कि इसके निर्देश अब भी `AGENTS.md` पर आधारित हैं। ऐप के दस्तावेज़ कहते हैं कि `custom instructions` बदलने से व्यक्तिगत निर्देश `AGENTS.md` में अपडेट होते हैं, और मुझे यही रिश्ता चाहिए:

- सुविधा के लिए ऐप settings
- टिकाऊपन के लिए फ़ाइल-आधारित निर्देश

इससे बाद में CLI के उपयोग को समझना भी आसान हो जाता है, क्योंकि वही आधारभूत निर्देश आगे तक साथ चलते हैं।

## रिपॉज़िटरी `AGENTS.md` अब भी ज़रूरी है, बस कहानी का केंद्र वही नहीं है

मैं नहीं चाहता कि यह लेख `AGENTS.md` nesting पर पूरा पाठ बन जाए, जबकि Codex इसे अच्छी तरह संभालता है।

मेरा तरीका इससे सरल है:

- व्यक्तिगत `AGENTS.md` Codex को मेरा बुनियादी व्यवहार देता है
- रिपॉज़िटरी `AGENTS.md` Codex को रिपॉज़िटरी से जुड़ी अपेक्षाएं देता है
- nested files सिर्फ उन दुर्लभ जगहों के लिए हैं जहां उनकी सच में जरूरत हो

इससे पूरी व्यवस्था समझने में आसान रहती है।

अगर मैं कोई अनजान रिपॉज़िटरी खोलूं और Codex अजीब बर्ताव करे, तो मैं वजह जल्दी समझना चाहता हूं। आम तौर पर उत्तर इनमें से एक होना चाहिए:

1. मेरे वैश्विक नियम पर्याप्त स्पष्ट नहीं हैं
2. रिपॉज़िटरी निर्देश गायब हैं
3. काम खुद बहुत व्यापक है

यह नहीं कि मुझे सोचना पड़े कि सात छिपी हुई निर्देश-परतों में इस बार किसने बाज़ी मार ली।

## मेरी Codex व्यवस्था में CLI की जगह

Mac ऐप मेरा मुख्य इंटरफ़ेस है। CLI कोई गौण वैकल्पिक सहारा नहीं है। वह वही Codex तंत्र है, बस अलग रूप में।

CLI कुछ वजहों से अब भी महत्वपूर्ण है:

- यह फ़ाइल-आधारित विन्यास को बहुत साफ बना देता है
- ठीक-ठीक व्यवहार को जांचना या script में बांधना इसमें आसान होता है

मैं CLI के लिए अलग नजरिया नहीं चाहता। मैं वही व्यक्तिगत `AGENTS.md`, वही रिपॉज़िटरी मार्गदर्शन, और दोनों में वही सीमाएं चाहता हूं।

यही निरंतरता उस बड़ी वजह का हिस्सा है जिसकी वजह से यह उत्पाद मुझे एकसार लगता है।

## अभी मेरी व्यावहारिक Codex व्यवस्था

अगर मैं आज Mac पर इसे बिल्कुल शुरुआत से तैयार करूं, तो क्रम यही रखूंगा।

### 1. सबसे पहले ऐप में Personal Instructions लिखें

`Cmd+,` से Codex ऐप settings खोलें, `Personalization` में जाएं, और custom instructions पहले वहीं लिखें। मैं अब भी `~/.codex/AGENTS.md` के हिसाब से सोचता हूं, लेकिन शुरुआती व्यवस्था और समीक्षा के लिए ऐप ही मेरी मुख्य जगह है।

### 2. व्यक्तिगत फ़ाइल छोटी और स्पष्ट रखें

परियोजना की संरचना यहां नहीं होनी चाहिए। यहां लंबे समय तक चलने वाले coding rules होने चाहिए:

- सख्त typing
- स्पष्ट त्रुटि-प्रबंधन
- चुपचाप लगने वाले fallback नहीं
- न्यूनतम बदलाव
- बिखरे हुए docs की जगह docstrings
- साफ terminal आदतें

### 3. Repository AGENTS.md सिर्फ रिपॉज़िटरी-सत्य के लिए जोड़ें

यहीं commands, architecture, constraints, naming, परीक्षण से जुड़ी अपेक्षाओं और संवेदनशील हिस्सों की जानकारी होनी चाहिए। यही रिपॉज़िटरी वाली परत है।

## अभी Codex मुझे आशाजनक क्यों लगता है

मैं अभी भी Codex के साथ शुरुआती दौर में हूं, इसलिए मैं इसे बढ़ा-चढ़ाकर पेश नहीं करूंगा।

लेकिन इसका मेल अभी से मजबूत लगता है:

- फ़ाइल पर आधारित निर्देशों की एक असली परत
- ऐप और CLI, जो एक-दूसरे से जुड़े लगते हैं, टकराते हुए नहीं

मेरे लिए इसे इस्तेमाल करते रहने के लिए इतना काफी है।

अब तक मेरे लिए सबसे अच्छी साबित हुई व्यवस्था सबसे कम चमकदार भी है: ठोस स्थायी निर्देश लिखो, रिपॉज़िटरी मार्गदर्शन को अलग रखो, और एजेंट को ऐसे आधार से शुरू होने दो जो पहले से तुम्हारे काम करने के तरीके से मेल खाता हो।

ढर्रा Cursor जैसा ही है। Claude Code जैसा ही है। उत्पाद अलग है, सीख वही है: सत्र तब बेहतर होता है, जब एजेंट यह अंदाज़ा लगाना बंद कर दे कि आप कौन हैं।
